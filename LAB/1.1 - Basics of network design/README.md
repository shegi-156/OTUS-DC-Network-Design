## Формирование hostname
Имя сетевого элемента должно однозначно идентифициоравть его тип, выполняемую функцию и принадлежность к локации

Предпологается, что hostname имеет вид: **[DC+POD]-[NE_T]-[NE_R+ID]**

где

**DC** - наименование ЦОД

**POD** - номер POD в пределах ЦОД

**NE_T** - тип сетевого элемента

**NE_R** - функциональная роль сетевого элемента

**ID** - сквозной номер сетевого элемента данного типа внутри POD

---

*Пример_1:* 

*hostname __MSK1-SW3-LEAF1__*

- *Датацентр - MSK (Москва)*
- *Номер POD`а - 1*
- *Тип сетевого элемента - SW3 (коммутатор уровня 3)*
- *Роль сетевого элемента - LEAF*
- *Номер сетевого элемента типа LEAF в данном POD - 1*

---

*Пример_2:*

*hostname __NSK3-SW3-SPINE2__*

- *Датацентр - Новосибирск*
- *Номер POD`а - 3*
- *Тип сетевого элемента - SW3 (коммутатор уровня 3)*
- *Роль сетевого элемента - SPINE*
- *Номер сетевого элемента типа SPINE в данном POD - 2*



## Распределения адресного пространства

>Не сторонник применять мнемонические правила для адресации, из личной практики это приводит к большому расходу подсетей при малой утилизации ip адресов. Как по мне проще заглянуть в IPAM и выделять подсети по порядку.

### Managment (OOB) 

> Для OOB управления оборудованием в стойках как правило используется отдельные коммутаторы в которые подключаются MGMT порты коммутаторов/маршрутизаторов и BMC серверных платформ (iLO/iDRAC/IPMI)

Для лабораторной работы выделен префикс **172.16.0.0/24**


### Loopback
Для лабораторной работы выделен префикс **10.0.0.0/24**


### P2P
Для лабораторной работы выделен префикс **10.0.1.0/24**
